# 3. Rust + Tauri 前提のプロジェクト構成案

ここは結論から言うと、**Rust + Tauri v2 + React (TypeScript)** 構成で進めます。

## 3.1 なぜ Rust + Tauri か

このアプリで必要になるのは以下です。

- オーバーレイ表示
- グローバルホットキー
- タスクトレイ
- ネイティブAPI連携（Windows API）
- ウィンドウ制御
- キャプチャUIの細かい制御
- 高速な画像処理
- 小さいバイナリサイズ・低メモリ消費

こういう **Windowsデスクトップ感の強いアプリ** において、\
**Tauri** は以下の利点があります。

- Webフロントエンド（React）でUIを高速に構築
- Rustバックエンドでネイティブ機能を安全かつ高速に実装
- Electron比で圧倒的に軽量
- セキュリティ面でも有利（デフォルトでサンドボックス）

### 採用方針

- **バックエンド:** Rust（Tauri v2）
- **フロントエンド:** React + TypeScript（Vite）
- **状態管理:** Zustand または React Context
- **スタイリング:** Tailwind CSS または CSS Modules

## 3.3 レイヤ責務

### フロントエンド（React / TypeScript）

役割:

- UI描画・画面遷移
- ユーザー操作の受付
- Tauri Commands の呼び出し
- フロントエンド状態管理

主な技術栈:

- **React** — UIコンポーネント
- **TypeScript** — 型安全なフロントエンド開発
- **Zustand** — 状態管理（軽量Store）
- **Tailwind CSS** — スタイリング
- **@tauri-apps/api** — Tauri JS API

### commands /（Tauri Command ハンドラ）

役割:

- フロントエンドからのリクエスト受付
- Service層への委譲
- レスポンス整形

例:

```rust
#[tauri::command]
async fn capture_full_screen(state: State<AppState>) -> Result<WorkspaceItem, AppError> {
    let service = &state.capture_service;
    let result = service.capture_full_screen().await?;
    Ok(result)
}

#[tauri::command]
async fn get_workspace_items(state: State<AppState>) -> Result<Vec<WorkspaceItem>, AppError> {
    let repo = &state.workspace_repo;
    let items = repo.get_all().await?;
    Ok(items)
}
```

### services /（ビジネスロジック）

役割:

- キャプチャ処理
- サムネイル生成
- 画像編集
- クリップボード操作
- ホットキー管理

例:

```rust
pub trait CaptureService: Send + Sync {
    fn capture_full_screen(&self) -> Result<CaptureResult, AppError>;
    fn capture_region(&self, region: Region) -> Result<CaptureResult, AppError>;
    fn capture_window(&self, hwnd: HWND) -> Result<CaptureResult, AppError>;
}
```

### repositories /（データアクセス）

役割:

- SQLite CRUD
- クエリ構築
- トランザクション管理

例:


### db /（データベース管理）

役割:

- SQLite接続管理
- マイグレーション実行
- コネクションプール

### storage /（ファイル管理）

役割:

- ファイル保存・読み込み
- ディレクトリ管理
- サムネイル保存

### platform /（OS固有処理）

役割:

- Windows API連携（FFI）
- スクリーンキャプチャの低レベル処理
- ウィンドウ列挙
- グローバルホットキー登録

### error.rs（共通エラー）

役割:

- アプリ全体のエラー型定義
- `thiserror` によるエラー変換

例:

```rust
#[derive(Debug, thiserror::Error)]
pub enum AppError {
    #[error("Database error: {0}")]
    Database(#[from] rusqlite::Error),
    #[error("IO error: {0}")]
    Io(#[from] std::io::Error),
    #[error("Capture error: {0}")]
    Capture(String),
}
```

## 3.6 技術選定メモ

### UI フレームワーク

- **Tauri v2**
  - Rustバックエンド + Webフロントエンドのハイブリッド
  - 軽量・高速・セキュア
  - ネイティブAPIへのアクセスが可能

### フロントエンド

- **React + TypeScript**
  - コンポーネントベースでUI構築しやすい
  - 型安全

### 画像処理

- **image** クレート（Rust）— 基本的な画像読み書き
- **Skia** バインディング または **Canvas API**（フロントエンド）— 注釈描画

### DB

- **SQLite**（`rusqlite` または `sqlx`）
  - ローカル完結に最適

### ログ

- `tracing` + `tracing-subscriber` を採用

### ホットキー・キャプチャ

- TauriのGlobalShortcut API
- Windows APIは `windows-rs` クレートで連携

### 動画

- 初期は後回し
- 入れるなら `FFmpeg` 連携 or Windows Capture API

## 3.7 MVPでの保存ポリシー案

### 画像

- 原本は `originals/`
- 編集済みは `edited/`
- サムネイルは `thumbnails/`

### 保存方式

初期は **非破壊保存を推奨**

- 元画像を残す
- 編集後は別ファイルで保存
- DBに current_path を持たせる

これにすると後から再編集や履歴追加がしやすいです。
