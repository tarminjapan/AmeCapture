## 実行ステップ（Workflow）

1. **Issueの現状確認**
   - Issue の内容と現在のソースコードを照らし合わせ、対応が完了しているか確認してください。
   - 既に対応済みであればその旨を報告して終了してください。未対応であれば次のステップへ進んでください。

2. **実装およびPull Requestの送信**
   - Issueの要件を満たす実装を行ってください。
   - **品質管理:** PR作成前に必ずテスト (cargo test) と静的解析 (cargo clippy) を実行し、エラーがないことを確認してください。
   - 実装完了後、適切なブランチ名 (例: feature/issue-ID) でPull Request (PR) を作成してください。
   - **重要:** PRには必ずIssueを関連付けてください。

3. **Gemini Code Assistによるレビュー対応ループ**
   - PR送信後、Gemini Code Assistからのレビューコメントを待機してください。
   - 指摘事項があった場合は、速やかに修正コードをプッシュ（再対応）してください。
   - 修正後、Gemini Code Assistのコメントに対して以下の定型文で返信してください：
     > `/gemini 対応しました。指摘事項は解決しましたか？`
   - この修正と確認のサイクルを、Gemini Code Assistから「問題なし（LGTM）」の旨の返答が得られるまで繰り返してください。

## 制約・ルール（Constraints）

- **自律性:** レビューの指摘内容を論理的に解釈し、最善の実装を選択してください。
- **継続性:** 全てのレビュー指摘が解決し、承認が得られるまでタスクを終了しないでください。ただし、解決が困難な場合は理由を添えて報告してください。
- **品質とセキュリティ:** セキュリティ脆弱性を避け、プロジェクトのコーディング規約やベストプラクティスを遵守してください。
